# Playground in Prod - Optimising Agents in Production Environments — Samuel Colvin, Pydantic

## Summary

Samuel Colvin is the creator of Pydantic (the Python validation library) and now runs Pydantic the company, which maintains Pydantic validation, Pydantic AI (agent framework), and Logfire (observability platform). He opens with a notable contrarian claim: "I don't really believe in AI observability — I think it's a feature, not a category. It will get eaten by either observability or AI at some point."

**The two tools being combined.** This workshop is a live demo of integrating Jepper with Logfire managed variables for end-to-end prompt optimization in production. Jepper is a genetic Pareto optimization library — it breeds prompts like racehorses, taking the best-performing candidates from the Pareto frontier, mixing them to generate new candidates, and iterating. Created by a first-year PhD student at Berkeley (Colvin notes the Python is not production-quality: not async, not well-typed). Logfire managed variables extends prompt management to any Pydantic-typed object — not just strings — stored in the platform and editable without code deploys.

**The optimization task.** Colvin built a Pydantic AI agent to extract political family relations (ancestor politicians) from Wikipedia pages for UK MPs, to answer a question for "The Rest Is Politics" podcast about how many MPs come from political dynasties (~24%, per his initial run). The agent struggles with a consistent failure mode: it includes spouses and children even when instructed to find only ancestors. This is a structured output task — schema: `list[PoliticalRelation]` — comparable to any extraction problem (invoices, addresses, fraud detection).

**Baseline vs. optimized performance.** Simple one-line prompt: 85–87% accuracy on the test set. Expert hand-crafted prompt (describing ancestor semantics more carefully): 92%. Jepper optimization over ~400 calls: 96%. Individual failure mode analysis in Logfire: Joe White case showed the initial prompt returning a spouse as a political relation; the expert prompt correctly excluded her. A four-times-great-grandfather who was "governor of Hudson Bay" (a public figure but not a politician) fooled both prompts.

**Shopify reference.** Colvin cites a production case: Shopify used Jepper to classify websites (fraud detection, tax category). Switching from full-page-to-GPT-5 naive approach to an agent with Qwen model + Jepper-optimized prompt dropped costs from ~$5M/year to ~$73K/year while improving performance.

**LLM-as-judge critique.** Colvin describes LLM-as-judge as "lunatics running the asylum" and recommends deterministic evals against golden datasets whenever possible. His golden dataset was itself generated by running Opus 4.6 on all MPs and hand-checking — a bootstrap approach for when ground truth doesn't exist.

**Managed variables as the production loop.** The end state Pydantic is building toward: Logfire detects production performance degradation, automatically triggers a Jepper optimization run against real production traces, proposes new prompt candidates via managed variables, and rolls them out without code deployment. The demo version requires manual steps; the fully automated version is in development.

## Why it matters
- Jepper as a prompt optimizer is a concrete tool with a Shopify benchmark ($5M → $73K) — directly citable with a cost number.
- The genetic Pareto breeding metaphor is the clearest explanation of how Jepper works and what distinguishes it from random search or hill-climbing.
- Colvin's "AI observability is a feature not a category" framing is a direct challenge to the observability-platform narrative — valuable for a chapter on the observability landscape.
- LLM-as-judge vs. deterministic evals: the "lunatics running the asylum" framing is quotable and backed by a working demo showing deterministic evals catching specific failure modes that a score wouldn't reveal.
